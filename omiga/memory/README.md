# 记忆系统核心公理与架构

> 本文档定义了 Omiga 记忆系统的基本原则和架构规范。所有记忆写入操作必须遵循这些公理。

---

## 核心公理 (Core Axioms)

### 公理 1：行动验证原则 (Action-Verified Only)

**定义**：任何写入 L1/L2/L3 的信息，必须源自**成功的工具调用结果**。

**执行标准**：
- ✅ 允许写入：`file_read` 成功返回内容、`shell` 执行成功返回输出、API 调用成功返回数据
- ❌ 禁止写入：模型推理的猜测、未执行的计划、假设性结论、失败的调用结果

**口号**：**No Execution, No Memory. (无行动，不记忆)**

**示例**：
```
✅ 正确：执行 `pip install requests` 成功后 → 写入 L2 "requests 已安装"
❌ 错误：模型认为 "requests 应该已安装" → 不写入
```

---

### 公理 2：神圣不可删改性 (Sanctity of Verified Data)

**定义**：经过行动验证的配置、避坑指南，在重构时**严禁丢弃**。

**执行标准**：
- 可以压缩文字、可以迁移层级（如 L2→L3）
- 绝不能丢失信息的准确性和可追溯性
- 记忆修改只能少量 patch，改不动宁愿不改

**示例**：
```
✅ 正确：将详细配置压缩为摘要，保留关键参数
❌ 错误：重构时删除了经过验证的避坑指南
```

---

### 公理 3：禁止存储易变状态 (No Volatile State)

**定义**：严禁存储随时间/会话高频变化的数据。

**禁止示例**：
- ❌ 当前时间戳
- ❌ 临时 Session ID
- ❌ 正在运行的 PID
- ❌ 绝对路径（除非是固定配置）
- ❌ 连接的设备信息

**允许示例**：
- ✅ 固定的文件路径（如 `~/.config/omiga/config.json`）
- ✅ 环境配置（如 `PYTHON=/usr/bin/python3.12`）
- ✅ API 端点（如 `https://api.example.com/v1`）

## 信息分类决策树

```
这条信息该放哪层？
    ↓
是『环境特异性事实』？
(IP、非标路径、凭证、ID、API 密钥等，大模型 Zero-shot 无法生成准确值)
    ├─ YES → L2 (facts.md 按 ## [SECTION] 组织)
    │         然后 → 按频率归入 L1 第一层 (key→value) 或第二层 (仅关键词)
    │
    └─ NO
         ↓
         是『通用操作规律』？
         (全局性避坑指南、排查方法、不针对特定任务的通用准则)
         ├─ YES → L1 [RULES] (仅限 1 句压缩准则)
         │
         └─ NO
              ↓
              是『特定任务技术』？
              (艰难尝试才能成功，且未来还能用到的任务特定技术)
              ├─ YES → L3 (专项 SOP 或脚本)
              │
              └─ NO → 判定为『通用常识』或『冗余信息』: 严禁存储，直接丢弃
```

## 记忆层级架构

```
┌─────────────────────────────────────────┐
│ L1: index.md (极简索引层 - 严格控制 ≤30 行) │
│    ↓ 导航指向 (Pointer)                  │
│ L2: facts.md (事实库层 - 可膨胀)          │
│    ↓ 详细引用 (Reference)                │
│ L3: *.md/*.py (记录库层 - 特定任务)        │
└─────────────────────────────────────────┘
```

---

### L1：全局记忆索引 (`index.md`)

**职责**：为 L2 和 L3 提供极简导航索引，确保关键能力可被发现。

**硬约束**：
- **行数限制**：≤ 30 行（强制）
- **内容结构**：
  - 第一层：高频场景 key→value（直接给出 SOP/py/L2 section 名）
  - 第二层：低频场景仅列关键词
  - RULES：红线规则 + 高频犯错点（压缩版）

**禁止**：
- ❌ 密码、API Key 等敏感信息
- ❌ "How to" 或详细解释
- ❌ 特定任务的技术细节
- ❌ 日志记录

**更新规则**：
- L2/L3 有新增/删除时，判断频率归入对应层
- 修改时只能少量 patch，不允许 overwrite 或 code run
- 改不动宁愿不改

---

### L2：全局事实库 (`facts.md`)

**职责**：存储全局环境性事实（路径、凭证、配置、常量等）。

**特征**：
- 趋势：随环境扩展而膨胀（可接受）
- 内容：按 `## [SECTION]` 组织的事实条目
- 同步：变化时更新 L1 的相应 TOPIC 导航行

**禁止**：
- ❌ 存储易变状态（见公理 3）
- ❌ 存储猜测/未验证信息（见公理 1）
- ❌ 存储大模型可推理的通用常识

---

### L3：任务级精简记录库 (`*.md` / `*.py`)

**职责**：补充 L1/L2 无法容纳、但对**特定任务**未来复用至关重要的少量详细信息。

**原则**：
- 只记录：跨会话仍重要、且难以通过少量 file_read / web_scan / 简单脚本快速重建的要点
- 优先写：该任务特有的隐藏前置条件、典型易踩坑点，一旦遗忘会导致高成本重试的信息
- 不记录：普通操作步骤、可在几步探测中重新获得的路径或状态信息

**形式**：
- **SOP** (`*_sop.md`)：为单一任务或小类任务保留极简的「关键前置 + 典型坑」清单，避免长篇教程
- **工具脚本** (`*.py`)：仅封装高复用、逻辑相对复杂且不希望每次都重新推理的处理流程

---

## L1 ↔ L2/L3 同步规则

| 操作 | L1 同步动作 |
|------|------------|
| L2/L3 新增场景 | 判断频率：高频→第一层加 key→value；低频→第二层加关键词 |
| L2/L3 删除场景 | 删除对应层的关键词/映射行 |
| L2/L3 修改值 | 若不影响场景定位则不动 L1 |
| 发现通用避坑规律 | 压缩为一句加入 RULES |

---

## SOP 生命周期

```
pending/ (待审查)
    ↓ (审查通过)
active/ (已批准，正在使用)
    ↓ (过期/被替代)
archived/ (历史记录)
    ↓ (审查拒绝)
rejected/ (被拒绝)

lessons/ (教训，从失败中提取)
```

---

## 记忆写入检查清单

在写入任何记忆之前，请确认：

- [ ] 信息源自成功的工具调用结果（公理 1）
- [ ] 不是易变状态（公理 3）
- [ ] 已确定正确的层级（决策树）
- [ ] L1 行数未超限（≤30 行）
- [ ] 不包含敏感信息（密码/API Key）
- [ ] 描述准确、可追溯
